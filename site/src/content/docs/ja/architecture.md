Arcnem Vision の中心にあるのは、画像解析のためのサーバーサイド基盤です。画像はワークフローAPIキー経由またはダッシュボード経由で取り込まれ、ワークフローはデータベース上の定義から読み込まれ、エージェントが MCP ツールを呼び出して解析し、その結果と実行履歴がダッシュボードに集約されます。

## 取り込み経路は2つある

### ワークフローAPIキー経路

ワークフローAPIキーは、組織とプロジェクトにひもづき、デフォルトワークフローを直接指します。クライアントや外部連携先は次の順で画像を送ります。

1. `/api/uploads/presign` を呼ぶ
2. S3互換ストレージへ直接アップロードする
3. `/api/uploads/ack` を呼ぶ

`ack` が来ると、API はアップロード済みオブジェクトを検証し、ドキュメントを作成し、`document/process.upload` を発火します。するとエージェントサービスが、そのワークフローキーに割り当てられたワークフローを読み込んで実行します。

### ダッシュボード経路

ダッシュボードには、運用担当者向けの別経路があります。

1. プロジェクトを選んで画像をアップロードする
2. ワークフローキーにひもづかないドキュメントを作る
3. 保存済みワークフローを任意に選んで実行する

この経路は、単発の確認、比較検証、再実行、運用中の調査に向いています。

## ワークフロー定義はコードではなくデータ

![パイプラインとスーパーバイザーワークフローを表示するワークフローライブラリ](/dashboard-workflows.png)

ワークフローエンジンは Postgres 上のデータとして管理されています。

- `agent_graphs`: 保存済みワークフロー
- `agent_graph_templates` と `agent_graph_template_versions`: 再利用可能な版付きテンプレート
- `agent_graph_nodes`: ノード定義
- `agent_graph_node_tools`: ツール割り当て
- `agent_graph_edges`: 明示的なグラフ接続

ダッシュボードのキャンバスは、これらのレコードを直接編集します。実行時にはエージェントサービスがグラフのスナップショットを読み込み、LangGraph の状態機械を組み立てます。

使えるノードタイプは4種類です。

- **worker**: モデル設定、プロンプト、任意の MCP ツールを持つ LLM ワーカー
- **tool**: 入出力マッピング付きの単一 MCP ツール呼び出し
- **supervisor**: 複数 worker を振り分ける LLM ベースの制御ノード
- **condition**: `contains` または `equals` で分岐する決定論的ノード

状態 reducer もグラフ定義に含められるため、実行中に各 state key を追記するのか上書きするのかも制御できます。

## MCP が解析レイヤーを担う

MCP サーバーは、ワークフロー実行とダッシュボードの検索・チャットの両方を支える解析レイヤーです。現在は次のツールを提供しています。

- ドキュメント説明文の生成
- 画像埋め込みの生成
- OCR 結果の生成
- セグメンテーション結果の生成
- 説明文埋め込みの生成
- 類似ドキュメントの検索
- 類似説明文の検索
- スコープ付きドキュメント検索
- スコープ付きドキュメント一覧
- 根拠付きドキュメント文脈の取得

これらのツールを使って、シードには次のようなワークフローが用意されています。

- `Document Processing Pipeline`
- `Image Quality Review`
- `Document Segmentation Showcase`
- `Semantic Document Segmentation Showcase`
- `OCR Keyword Condition Router`
- `OCR Review Supervisor`

## 元画像も派生結果も保存する

Arcnem Vision は「一回推論して終わり」の仕組みではありません。元画像と派生結果をまとめて保存し、後から検索・確認・再利用できる状態を作ります。

- `documents`: 元画像と派生画像
- `document_descriptions`: 保存済み説明文
- `document_embeddings`: 画像埋め込み
- `document_description_embeddings`: 説明文埋め込み
- `document_ocr_results`: OCR テキスト、メタデータ、生ペイロード
- `document_segmentations`: セグメンテーションのメタデータと派生ドキュメントへの参照

## 実行履歴と状態遷移を追える

すべての実行は次のテーブルで追跡されます。

- `agent_graph_runs`
- `agent_graph_run_steps`

記録されるのは、主に次の内容です。

- 初期状態
- 最終状態
- 各ステップの state 差分
- 開始時刻と終了時刻
- エラー

ダッシュボードは Server-Sent Events を購読しているため、ドキュメント作成、OCR保存、説明文更新、セグメンテーション作成、実行の作成、ステップ更新、完了をリアルタイムで確認できます。

## 検索と根拠付きチャット

Docs タブは、保存済みコーパスの上に成り立っています。

- 意味検索では保存済み埋め込みを使う
- 埋め込みがない場合は語彙検索にフォールバックする
- コレクションチャットは説明文、OCR、関連セグメントを根拠に回答する

会話が一時的なエージェント状態に依存しないので、運用画面として安定した検索・参照面になります。

## ダッシュボードが運用の中枢

ダッシュボードは補助UIではなく、プラットフォームの操作面そのものです。

- プロジェクトとワークフローキーを作る
- APIキーを発行・無効化・更新する
- ワークフローキーへワークフローを割り当てる
- ワークフローやテンプレートを編集する
- 単発の画像をアップロードする
- OCRやセグメンテーション結果を確認する
- コーパスを検索し、チャットで横断的に調べる
- 実行中・過去の実行を追う

## Flutter クライアントの位置づけ

Flutter アプリは、撮影、プレビュー、GenUI 実験を行うためのデモクライアントです。便利ではありますが、プロダクトの主役ではありません。価値の中心にあるのは、サーバーサイドのワークフロー基盤、ダッシュボードによる運用、そして永続化された解析パイプラインです。
