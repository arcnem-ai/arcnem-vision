ダッシュボード（`server/packages/dashboard`）は、Arcnem Vision の運用を担う中核画面です。Flutter アプリを開かなくても、ここだけでコア体験はほぼ完結します。プロジェクトやデバイスを管理し、ワークフローを編集し、画像を投入し、結果と実行の状態遷移まで確認できます。

![ダッシュボード — プロジェクト、デバイス、APIキー管理](/dashboard-projects.png)

## タブ概要

- **Projects & Devices**: プロジェクト作成、デバイス登録、デフォルトワークフローの割り当て、APIキーの発行・更新
- **Workflow Library**: グラフワークフローの作成と編集、テンプレートの参照、テンプレート起点での新規グラフ作成
- **Docs**: アップロード済み画像の閲覧、意味検索、コレクションチャット、ダッシュボードからの直接アップロード、OCR結果やセグメンテーション結果の確認、任意ワークフローの投入
- **Runs**: 実行の開始から完了までをリアルタイムで確認

## プロジェクト、デバイス、APIキー

1. プロジェクトを作成します。
2. そのプロジェクト内にデバイスを追加し、デフォルトで実行する保存済みワークフローを選びます。
3. デバイスが画像を送れるよう API キーを発行します。

補足:

- 生成されたシークレットは一度しか表示されません。以後は公開識別子のみが見えます。
- 既存キーは、デバイス本体を変えずに名前変更、無効化、削除ができます。
- ワークフロー割り当てはデバイス単位なので、1つのプロジェクト内で通常処理、OCRレビュー、品質レビュー、セグメンテーション用デバイスを混在できます。
- デバイス経由のアップロードは API キー経路を通り、`/uploads/ack` のあとにそのデバイスの割り当てワークフローが自動実行されます。

## ワークフロー作成

![ワークフローライブラリ](/dashboard-workflows.png)

**Workflow Library** では、空のワークフローを一から作ることも、再利用テンプレートを土台に始めることも、既存グラフを編集することもできます。

![テンプレートから新しいグラフを起動するためのテンプレートピッカー](/dashboard-template-picker.png)

### テンプレートを起点に始める

1. Workflow Library 上部の **Browse Templates** を開きます。
2. ワークフロー名、ノードの役割、ツール名で絞り込みます。
3. カードに表示されるバージョン、公開範囲、entry node、edge数、これまでにそのテンプレートから起動した件数、代表ノードを確認します。
4. **Use Template** を押します。
5. テンプレート内容が組織用の新規ワークフローとして複製され、キャンバスがそのグラフに切り替わります。ライブラリ上でも、どのテンプレートの何版から作ったかを確認できます。
6. 同名ワークフローが既にある場合は、末尾に連番を付けて保存名を自動調整します。

- 起動後のワークフローはテンプレート本体とは別物です。新しいグラフ側で node、edge、tool、metadata を編集しても、元テンプレートは変わりません。

### ノードタイプ

| ノードタイプ | 役割 | 必須設定 |
|---|---|---|
| `worker` | ReActスタイルのワーカーエージェント | モデル、任意のシステムメッセージ、任意のツール |
| `supervisor` | 複数ワーカーのオーケストレーション | モデル、`config.members`（workerノードキー） |
| `condition` | stateを見て分岐する決定論的ノード | `config.source_key`、`operator`、`value`、`true_target`、`false_target`、任意で`case_sensitive` |
| `tool` | 単一ツール呼び出しノード | ツール1つ、任意で入出力マッピング |

### ワーカーへのツール割り当て

1. `worker` ノードを選択します。
2. **Assigned tools** で1つ以上のツールを選びます。
3. ワークフローを保存します。

1つの worker に複数ツールを割り当てられます。

### ツールノードの入出力マッピング

`tool` ノードでは、ツールスキーマ項目をグラフ状態キーに対応付けます。

- `input_mapping`: グラフ状態 -> ツール入力
- `output_mapping`: ツール出力 -> グラフ状態

固定値を渡す場合は `_const:` を使えます（例: `_const:image/png`）。

### condition ノードの分岐設定

`condition` ノードは、LLM に判断させるほどではない単純な分岐に向いています。

- `source_key`: 判定対象のstateキー。例: `ocr_text`
- `operator`: `contains` または `equals`
- `value`: 比較する文字列
- `true_target` / `false_target`: 遷移先ノードキー、または `END`
- `case_sensitive`: 任意の真偽値
- `outputKey`: 任意。判定結果の true / false を後続ステップに残したいときに使います

condition ノードにはモデルやツールを設定しません。出力エッジも管理対象で、常に2本必要であり、`true_target` と `false_target` に対応している必要があります。

保存前には、ノードキーの一意性、モデル必須条件、tool ノードの単一ツール制約、supervisor メンバーの整合性、condition ノードの遷移先整合性、condition ノードごとの2本の管理エッジ、エントリーノードから `END` へ到達できるかどうかが検証されます。

セグメンテーション用フローも、仕組みとしては通常のワークフローと同じです。違うのは呼び出すツールだけで、バージョン付きセグメンテーションモデルをデータベースに登録し、MCP 経由で実行します。OCR も同様で、`create_document_ocr` を呼ぶワークフローとして組み立てますが、結果は派生画像ではなく元ドキュメントにひもづく抽出テキストとして保存されます。

## Docs: 検索、コレクションチャット、アップロード、OCR結果、セグメンテーション結果

![新しいシード画像を表示するDocsタブ](/dashboard-docs.png)

- 意味検索では、まずセマンティック検索を試し、埋め込みがなければ語彙検索にフォールバックします。
- **Ask The Collection** を開くと、組織内のドキュメント群を横断して質問できます。回答は説明文、OCRテキスト、関連セグメントを根拠に組み立てられます。
- 回答の下には出典カードが並び、どの資料を根拠にしたかをプロジェクト名やデバイス名つきで追えます。
- **New chat** は、その場の会話履歴だけをリセットします。画像一覧や検索状態はそのままです。
- **Add From Dashboard** は、デバイス自動処理とは別の経路です。画像をまずドキュメントとして保存し、その後で好きなワークフローを選んで実行できます。
- ドキュメントをクリックすると、元デバイスの保存済み割り当てを変えずに別ワークフローを投入できます。

![コレクションチャットを開いたDocsタブ](/dashboard-docs-chat.png)

![関連セグメンテーション結果を持つ選択済みドキュメント](/dashboard-docs-segmentation-detail.png)

- OCR結果は元ドキュメントにひもづいたまま保持され、使用モデル、抽出テキスト、取得時刻、平均信頼度、正規化後のペイロードを確認できます。
- 派生したセグメント画像は元画像の下にまとまって表示され、使用モデルやプロンプトも確認できます。
- セグメンテーション結果は通常のドキュメントとして保存されるため、後続ワークフローでも再利用できます。
- OCR結果は別ドキュメントを作らず、`document_ocr_results` に保存されるため、同じ画像に対する抽出履歴をその場で追えます。

## Runs とリアルタイム更新

![実行詳細を展開したRunsタブ](/dashboard-run-detail.png)

- ダッシュボードは `/api/realtime/dashboard` を Server-Sent Events で購読します。
- **Docs** はドキュメント作成、OCR結果保存、説明文生成、セグメンテーション保存時に更新されます。
- **Runs** は実行の作成、ステップ変化、完了に合わせて更新されます。
- 実行を展開すると、初期状態、各ステップの state 差分、最終状態、実行時間、エラーを確認できます。

## OCRとセグメンテーションのワークフロー

- `create_document_ocr` は、正規化済みテキスト、生の OCR ペイロード、平均信頼度をドキュメントに保存します。
- OCR の振り分けが単純なルールで済むなら `condition` ノードが向いています。例: 「URGENT を含むか」。
- OCR の振り分けに文脈判断が必要なら `supervisor` が向いています。例: 請求書系か運用メモ系か。
- `models` テーブル上のバージョン付きモデルをセグメンテーションモデルとして登録し、MCP から呼び出せます。
- `create_document_segmentation` は生の結果ペイロードと派生セグメント画像の両方を保存します。
- シードには、`OCR Keyword Condition Router`、`OCR Review Supervisor`、言語セグメンテーション、セマンティックセグメンテーションのショーケース用ワークフローと、それらを元にした再利用テンプレートが含まれます。空のキャンバスからの編集も、テンプレート起点の作成もすぐ試せます。
